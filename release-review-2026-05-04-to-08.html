The Cascade Worth Talking About

Fix exposed pre-existing latent bugs — new errors in 47 minutes

The single most useful pattern from this week. Fixing a v1.89.11 regression in v1.89.14 unmasked three latent crash classes in the same file, all of which had been hidden by the silent-failure mode of the original defect.

v1.89.11 (Mon) — PR #4716 silently removes TransferFilter::fromArray(). Endpoint returns 200 with empty filters. Latent bugs in fromArray() never reached because parsing didn't run.
2 days later (Wed) — Alice notices the transfers list is “out of order” and filters don’t work. Filed as #4780.
v1.89.14 (Wed 19:19) — PR #4781 restores the ::fromArray() call. Filter parsing path active again.
v1.89.14 + 47 min (Wed 20:06) — KONVOY-API-2V6 fires (Undefined array key “sortBy”). Followed 9 minutes later by KONVOY-API-2V7 (TypeError: filter null) and 2V8 (ValueError: BOGUS enum). Tracked under #4793.
v1.89.15 (Fri 02:30) — PR #4795 hardens both extractFilter() (coerce non-array to []) and fromArray() (isset+is_array guards, tryFrom for enum, null-column guard for Sort). Adds 3 functional tests. Covers 2V6/V7/V8 plus 2VC/VD/VE that fired Thu afternoon.

Worth being precise about in the postmortem: the three Sentry incidents are genuinely caused by v1.89.14 (the fix unmasked them), but the underlying defects are pre-existing — they predate v1.89.11. This distinction matters when we frame “what to do next time.”

The Fix That Was Rolled Back

#4794 · NetSuite postingPeriod 400s · PR #4778 reverted

Cleanest example this week of “a fix that was an issue, not a fix.”

  1. #4777 — Mariana (Finance) reports NS invoice postingPeriod uses server’s current month instead of order’s completedDate.
  2. PR #4778 in v1.89.14 changes getPostingPeriod() to take order’s completedDate.
  3. Thu morning: 3 Sentry incidents fire (KONVOY-API-2V9/2VA/2VB) when invoices for orders completed today fall in closed/locked NS periods.
  4. Rob drafts PR #4796 with a closed-period fallback.
  5. Finance follow-up later that day: Mariana’s concern was actually about NS tranDate, not postingPeriod. Finance has a known manual adjustment workflow for backdated transactions and specifically does not want Konvoy to derive postingPeriod from completedDate. The original behaviour was correct.
  6. PR #4796 abandoned. PR #4778 reverted separately. #4794 closed as not planned.
Lesson saved (Rob, in the issue close comment):
“We should explicitly confirm with the domain owner which specific NetSuite field their concern relates to (tranDate vs postingPeriod vs subsidiary etc.) before changing integration code. NetSuite has many overlapping date/period/account fields and the wrong-field assumption is invisible until it’s deployed.”

The frontend completedDate editability concern (Mariana’s actual underlying worry) is correctly tracked in #4787 and #4788.

Not Konvoy at All

#4775 · NetSuite custom-list lookup 500s · OPEN P1

Reads as a Konvoy regression in Sentry but is a NetSuite-side change. First seen 2026-05-05 20:33 UTC — Wed 6 May 06:33 AEST — before v1.89.13 deployed at 09:22.

Two plausible upstream causes: (1) CUSTOMLISTKV_CUSTOMER_REGION_LIST custom list was renamed/deleted in NS-OC, or (2) NetSuite has tightened SuiteQL identifier resolution (script-id support has been dropping in their releases). Either way, the Konvoy-side code at Traits/Location.php:27-34 hasn’t changed.

Konvoy-side fragility worth fixing regardless: the try { … } catch (NetSuiteException) { throw new NetSuiteException(...); } at NetSuiteService.php:101-107 turns an optional region-attribute lookup failure into a fatal 500 on the whole organisation PATCH. The success branch already has the right policy (if count===0 → omit field); the catch branch should mirror it. Hotfix Part A this morning unblocks Alice on prod-oc.

Open Questions for the Weekend Postmortem

These are the questions the postmortem itself should answer. They’re framed as discussion prompts — not as conclusions to validate.

QUESTION 1 · SCOPE

What is our threshold for “a release is too big”?

v1.89.11 had 50+ items spanning FL backend, MCP OAuth, OrderForm rewrite, Symfony plumbing, dependency bumps, and frontend redesigns. Of the issues filed this week, only ~6 trace to it as genuine regressions — but the cascade pattern suggests review necessarily got superficial at that scope.

cadencebatchingPR review
QUESTION 2 · PROCESS

How should we treat same-day re-deploys?

v1.89.14 deployed at 19:19 AEST. KONVOY-API-2V6 fired at 20:06 — 47 minutes later. We had no monitoring window between deploys. Should there be a 30-minute “watch the dashboard” hold, or a Sentry alert that fires on any new unresolved issue first-seen within 1 hour of a deploy tag?

deploymonitoringSentry rules
QUESTION 3 · INTEGRATION

What’s our process for NetSuite-touching PRs?

The #4794 rollback was caused by ambiguity over which NS field Mariana’s concern related to (tranDate vs postingPeriod vs subsidiary). Should the PR template require “specify the NS field by exact name + which Finance role owns it”? Should we maintain a 1-page glossary of the NS fields we touch alongside the integration code?

NetSuitedomain ownershipPR template
QUESTION 4 · SILENT FAILURES

Where else are we relying on silent-failure modes that hide defects?

Three of this week’s thorniest issues came from the same shape: Symfony silently auto-wired an empty TransferFilter; #[MapRequestPayload] silently dropped an unknown JSON key for 2.5 months; the new OrderForm silently overwrote saved overrides with location defaults. Issue #4789 (ALLOW_EXTRA_ATTRIBUTES = false) is one fix; what else needs the same treatment?

silent failuredefensive codetype safety
QUESTION 5 · SENTRY HYGIENE

Why are 11 of this week’s Sentry IDs still unresolved?

The defects are fixed and shipped, but the team isn’t marking the Sentry issues resolved post-deploy. The signal is muddied: we can’t tell which open Sentry IDs are real and which are stale. Auto-resolve via Sentry GitHub integration on commit-message Fixes KONVOY-API-XXX? Deploy-checklist tickbox? Both?

Sentryworkflowdeploy checklist
QUESTION 6 · COVERAGE

How did #4790 (silent pricing overwrite) reach production?

P1 silent data loss into invoice flow. The legacy OrderForm had the correct merge logic; the new one regressed it. No test caught the “saved override should beat location default” invariant. What test would have caught it — and is it the kind of test we’d realistically write before someone reports the bug?

testingdata integrityQA
QUESTION 7 · ATTRIBUTION

Was #4744 (clone-date) actually a v1.89.11 regression?

Rob’s root-cause note flagged it as “possibly latent — PR #4662 may have only exposed the bug rather than caused it.” The open question to Marc/Alice was whether clone-with-blank-date was verifiably working in v1.89.10. Answer that, and we either tag this for a regression test on the #4662 path or close out the question with “latent, fixed, not a regression.”
